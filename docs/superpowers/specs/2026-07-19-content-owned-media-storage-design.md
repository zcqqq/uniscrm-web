# Content-Owned Media Storage Design

## Context

The TikTok photo-post feature ([2026-07-16-tiktok-photo-post-ai-image-design.md](2026-07-16-tiktok-photo-post-ai-image-design.md)) introduced `link`'s `MEDIA_BUCKET` R2 binding: `content` generates image bytes and returns them over an HTTP response, `link` receives those bytes and writes them into its own R2 bucket, then serves them back via an unauthenticated `GET /public/media/:key` route so TikTok's `PULL_FROM_URL` publish mode can fetch them.

This design relocates that bucket and route to `content` — the module that actually generates the media should own storage for what it produces, rather than shipping bytes to `link` purely to be stored. This is a prerequisite for an upcoming video-processing feature ("Video Action: Add Subtitle") whose output artifacts (potentially much larger than the current images) will also need R2 storage; centralizing ownership in `content` now avoids splitting media storage across two modules for no reason, and avoids ever needing to ship large video files over an HTTP body between two Workers.

There is no production data in the current bucket, which removes the usual caution against touching shipped storage infrastructure.

## Scope

- New R2 bucket `uniscrm-content-media-dev` (dev) / `uniscrm-content-media` (prod), bound as `MEDIA_BUCKET` in `content/wrangler.toml`.
- New `CONTENT_URL` env var in `content/wrangler.toml` (mirrors `link`'s existing `LINK_URL` — content needs to know its own public base URL to construct served-media links).
- New `GET /public/media/:key` route in `content/src/index.ts`, unauthenticated (same as the original), registered so it isn't shadowed by the `fetch` wrapper's HTML-redirect/asset-serving logic — following the existing `/health` route's precedent for an unauthenticated path outside `/api/*`/`/internal/*`.
- `content/src/routes-internal.ts`'s `/internal/generate-image` route's response contract changes: instead of returning raw image bytes, it writes the generated bytes directly into its own `MEDIA_BUCKET` and returns `{ url: string }`.
- `link/src/routes-internal.ts`'s TikTok photo-post route simplifies: no longer reads `res.arrayBuffer()` or calls `MEDIA_BUCKET.put()` — it reads `{ url }` from `content`'s JSON response and uses that URL directly in `photoUrls`.
- `link`'s `MEDIA_BUCKET` binding, its `GET /public/media/:key` route, and its `[[r2_buckets]]` `wrangler.toml` blocks are all removed.
- The old R2 buckets (`uniscrm-link-media-dev`/`uniscrm-link-media`) are deleted once the migration is verified working — an explicit, controller-executed step (not delegated to a subagent) given it's a destructive action against real Cloudflare infrastructure, reconfirmed immediately before execution regardless of this design's stated decision.

## Out of scope

- Re-verifying TikTok's `PULL_FROM_URL` domain requirement against `content`'s domain (`content-dev.uni-scrm.com`/`content.uni-scrm.com`) — this was already an outstanding, never-completed manual step against `link`'s domain before this migration; it remains outstanding, just against a different domain now. Requires the user's own TikTok developer console access.
- Any change to the TikTok photo-post feature's actual behavior, pricing, image count, or provider selection — this is a pure storage-ownership relocation, not a feature change.
- The upcoming "Video Action: Add Subtitle" feature itself — this spec only prepares the storage foundation it will use; that feature gets its own separate spec and plan.
- Updating `link/tests/services/tiktok-publish.test.ts`'s literal example fixture URLs (`https://link-dev.uni-scrm.com/public/media/a`, etc.) — these are synthetic strings in unit tests of the TikTok publish request-body builder, not derived from the real route or domain; changing them serves no functional purpose.

## 1. `content` module: new bucket, new URL var, new serving route

**`content/wrangler.toml`**: add, in both `[env.dev]` and `[env.production]` sections:

```toml
[[env.dev.r2_buckets]]
binding = "MEDIA_BUCKET"
bucket_name = "uniscrm-content-media-dev"
```

```toml
[[env.production.r2_buckets]]
binding = "MEDIA_BUCKET"
bucket_name = "uniscrm-content-media"
```

Add `CONTENT_URL` to each env's `[env.X.vars]` block:
- dev: `CONTENT_URL = "https://content-dev.uni-scrm.com"`
- production: `CONTENT_URL = "https://content.uni-scrm.com"`

**`content/src/types.ts`**: add to the `Env` interface:

```ts
  MEDIA_BUCKET: R2Bucket;
  CONTENT_URL: string;
```

**`content/src/index.ts`**: add the serving route, mirroring `link`'s original exactly:

```ts
app.get("/public/media/:key", async (c) => {
  const object = await c.env.MEDIA_BUCKET.get(c.req.param("key"));
  if (!object) return c.notFound();
  return new Response(object.body, {
    status: 200,
    headers: { "Content-Type": object.httpMetadata?.contentType || "application/octet-stream" },
  });
});
```

Place this registration before the `fetch` wrapper's HTML-redirect/asset-serving branch takes effect — the existing `fetch` handler (lines 104-135 today) checks `!url.pathname.startsWith("/api") && !url.pathname.startsWith("/internal")` (and a similar check gating the ASSETS-first branch) to decide whether to redirect an unauthenticated browser request to `${WEB_URL}/login` or serve static assets before ever reaching `app.fetch`. `/health` is already exempted from the ASSETS-first branch by an explicit check; `/public` must be added to that same exemption (both places in the wrapper that currently special-case `/health`), otherwise a request to `/public/media/:key` would be swallowed by the wrapper before Hono's router ever sees it.

## 2. `content/src/routes-internal.ts`: `/internal/generate-image` contract change

Current behavior: generates image bytes via `generateImage(c.env, params)`, returns them as the raw HTTP response body.

New behavior: after generating the bytes, write them into `c.env.MEDIA_BUCKET` under a random key and return the resulting URL as JSON:

```ts
router.post("/generate-image", async (c) => {
  const { tenantId, prompt, provider, skillId } = await c.req.json<{
    tenantId: number;
    prompt: string;
    provider: "default" | "openai";
    skillId?: string;
  }>();

  if (!tenantId || !prompt || !provider) {
    return c.json({ error: "tenantId, prompt, provider required" }, 400);
  }

  try {
    const { bytes, contentType } = await generateImage(c.env, { tenantId, prompt, provider, skillId });
    const key = crypto.randomUUID();
    await c.env.MEDIA_BUCKET.put(key, bytes, { httpMetadata: { contentType } });
    return c.json({ url: `${c.env.CONTENT_URL}/public/media/${key}` });
  } catch (err) {
    console.error(JSON.stringify({ event: "generate_image_failed", tenantId, provider, error: String(err) }));
    return c.json({ error: "Image generation failed" }, 502);
  }
});
```

## 3. `link/src/routes-internal.ts`: TikTok photo-post simplification

Current (per-image loop):

```ts
for (let i = 0; i < requestedCount; i++) {
  const res = await fetch(`${c.env.CONTENT_URL}/internal/generate-image`, {
    method: "POST",
    headers: { "Content-Type": "application/json", "X-Internal-Secret": c.env.INTERNAL_SECRET },
    body: JSON.stringify({ tenantId, prompt: prompts.message_image, provider: imageProvider, skillId: imageSkillId }),
  });
  if (!res.ok) continue;
  const bytes = await res.arrayBuffer();
  const contentType = res.headers.get("Content-Type") || "image/jpeg";
  const key = crypto.randomUUID();
  await c.env.MEDIA_BUCKET.put(key, bytes, { httpMetadata: { contentType } });
  photoUrls.push(`${c.env.LINK_URL}/public/media/${key}`);
}
```

New:

```ts
for (let i = 0; i < requestedCount; i++) {
  const res = await fetch(`${c.env.CONTENT_URL}/internal/generate-image`, {
    method: "POST",
    headers: { "Content-Type": "application/json", "X-Internal-Secret": c.env.INTERNAL_SECRET },
    body: JSON.stringify({ tenantId, prompt: prompts.message_image, provider: imageProvider, skillId: imageSkillId }),
  });
  if (!res.ok) continue;
  const body = await res.json() as { url: string };
  photoUrls.push(body.url);
}
```

Best-effort semantics (partial image failures don't fail the whole post) are unchanged — only the per-image mechanics (bytes+store vs. direct URL) change.

## 4. `link` cleanup

Remove entirely:
- `link/wrangler.toml`'s two `[[r2_buckets]]` blocks (lines 53-55, 129-131 today).
- `link/src/types.ts`'s `MEDIA_BUCKET: R2Bucket;` entry.
- `link/src/index.ts`'s `GET /public/media/:key` route.
- `link/tests/services/public-media.test.ts` (superseded by an equivalent new test in `content`).
- `mockMediaBucket()` and its use from `link/tests/services/routes-internal-tiktok-photo-post.test.ts` (link's env no longer has this binding at all).

## 5. Old bucket deletion

Once the migration is deployed and verified (see Verification below), delete the old buckets:

```bash
wrangler r2 bucket delete uniscrm-link-media-dev
wrangler r2 bucket delete uniscrm-link-media
```

This step is executed directly by the controller during implementation, not delegated to a subagent, and is reconfirmed immediately before running regardless of this design's already-recorded decision to delete — per this project's standing caution around destructive actions against real cloud infrastructure.

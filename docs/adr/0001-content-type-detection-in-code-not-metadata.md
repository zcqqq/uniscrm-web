# Conditional content_type detection stays in poller code, not in PropMapping

X Articles arrive as a tweet payload with an extra `article` structure (see `_reference/x/post.json`); we need `content_type` to be `ARTICLE` instead of the default `TWEET` when that structure is present. PropMapping (the declarative model in `metadata/*.ts`) only supports a fixed `value` or a `dataId` extraction path — it has no conditional/presence-based logic, and extending it for a single use case would add a generic mechanism ahead of any second consumer. We chose to special-case this in `x-posts.ts`'s poller (`if (item.article) props.content_type = "ARTICLE"`) after `resolveProps` runs, keeping `metadata/*.ts` purely declarative. If a second conditional-detection case appears elsewhere, revisit generalizing PropMapping instead of adding a second special case.
